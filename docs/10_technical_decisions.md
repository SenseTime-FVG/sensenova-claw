# 技术决策记录

## 概述

本文档记录了 AgentOS v0.1 版本开发过程中的关键技术决策及其理由。

## 前端技术选型

### Next.js + TypeScript

**决策**: 使用 Next.js 14 (App Router) + TypeScript 构建前端

**理由**:
- Next.js 提供优秀的开发体验和性能优化
- App Router 支持服务端组件，减少客户端 JavaScript
- TypeScript 提供类型安全，减少运行时错误
- 生态系统成熟，社区活跃

### React Context API

**决策**: 使用 React Context API 而非 Redux/Zustand

**理由**:
- v0.1 版本状态管理需求简单
- Context API 足够满足需求，无需引入额外依赖
- 减少学习成本和代码复杂度
- 后续可根据需要迁移到更强大的状态管理方案

### 原生 WebSocket

**决策**: 使用原生 WebSocket 而非 Socket.IO

**理由**:
- 需求简单，不需要 Socket.IO 的高级特性（房间、命名空间等）
- 减少依赖和打包体积
- 原生 API 性能更好
- 浏览器支持度已经足够

### shadcn/ui + Tailwind CSS

**决策**: 使用 shadcn/ui 组件库配合 Tailwind CSS

**理由**:
- shadcn/ui 提供无依赖的组件代码，可完全自定义
- Tailwind CSS 提供灵活的样式系统
- 易于实现 VS Code 风格的深色主题
- 组件质量高，可访问性好

## 后端技术选型

### FastAPI + Python 3.12

**决策**: 使用 FastAPI 框架和 Python 3.12

**理由**:
- FastAPI 原生支持异步和 WebSocket
- 自动生成 API 文档
- 类型提示和数据验证（Pydantic）
- Python 3.12 性能提升显著
- AI/ML 生态系统丰富

### asyncio

**决策**: 使用 asyncio 而非 trio

**理由**:
- asyncio 是 Python 标准库，无需额外依赖
- 生态系统更成熟，第三方库支持更好
- 团队更熟悉 asyncio
- 性能满足需求

### 内存队列事件总线

**决策**: v0.1 使用内存队列，不使用 Redis

**理由**:
- 简化部署，无需额外服务
- 单机性能足够
- 降低开发和运维复杂度
- 后续可平滑迁移到 Redis

### SQLite

**决策**: 使用 SQLite 而非 PostgreSQL/MySQL

**理由**:
- 零配置，无需独立数据库服务
- 适合本地开发和单用户场景
- 性能满足需求
- 数据文件便于备份和迁移

## 架构设计决策

### 事件驱动架构

**决策**: 采用事件驱动而非传统的循环驱动

**理由**:
- 更好的解耦，模块间依赖更少
- 易于追踪和调试（所有事件都有记录）
- 支持异步并发执行
- 易于扩展新功能
- 便于实现可观测性

### 双总线机制

**决策**: 使用 Public Bus + Private Bus 的双总线设计

**理由**:
- 保证会话隔离，避免事件串扰
- 支持多 Agent 并发运行
- 便于实现细粒度的事件过滤
- 为未来的分布式部署预留空间

### 非流式响应

**决策**: v0.1 版本不实现流式响应

**理由**:
- 简化实现复杂度
- 降低前后端协调难度
- 快速验证核心功能
- 后续版本可轻松添加

## 安全决策

### 无沙箱执行

**决策**: v0.1 版本工具直接执行，不使用沙箱

**理由**:
- 简化开发，快速迭代
- 本地开发环境风险可控
- 明确标注为开发版本
- 生产版本会增强安全措施

**风险**: 恶意代码可能造成系统损害

**缓解措施**:
- 文档中明确警告
- 仅用于可信环境
- 后续版本实现沙箱

### 全系统文件访问

**决策**: v0.1 版本不限制文件访问范围

**理由**:
- 提供最大灵活性
- 简化权限管理
- 适合个人开发环境

**风险**: 可能误删重要文件

**缓解措施**:
- 用户需自行备份重要数据
- 后续版本添加权限控制

## 性能决策

### 并发工具执行

**决策**: 多个工具调用并发执行

**理由**:
- 显著提升执行效率
- 充分利用异步特性
- 用户体验更好

### 统一 15 秒超时

**决策**: 所有工具统一 15 秒超时

**理由**:
- 简化配置
- 避免长时间阻塞
- 大多数操作可在 15 秒内完成
- 后续可根据工具类型细化

## 未来考虑

### 待实现的功能

以下功能在 v0.1 中暂不实现，但架构已预留扩展空间：

1. **流式响应**: 提升用户体验
2. **Token 管理**: 成本控制和限流
3. **多模型切换**: 用户可选择不同模型
4. **沙箱执行**: 提升安全性
5. **分布式事件总线**: 支持多实例部署
6. **用户认证**: 多用户支持
7. **上下文压缩**: 处理超长对话
8. **插件系统**: 第三方扩展

需要在后续版本中解决的问题：

1. 错误处理不够完善
2. 缺少完整的单元测试
3. 日志系统需要优化
4. 性能监控和指标收集
5. 配置验证不够严格
