# Research Union 设计文档

## 1. 背景与目标

当前 `research-union` 已具备“先确认方案、再主链搜索、不足时补充来源、最后输出报告”的基础能力，但仍存在几个问题：

- 没有明确区分“简单搜索”与“复杂调研”，容易让简单请求也被重流程拖慢
- `outline`、来源策略、执行阶段的边界还不够清晰
- 复杂请求的确认产物没有标准化结构，不利于后续执行、日志排查和多 agent 协作
- `union-search-plus` 已有明确的“补充来源”职责，需要在上层 skill 中进一步固定边界

本设计的目标是：

- 为 `research-union` 增加“简单 / 复杂”双路径
- 为复杂请求建立稳定的五阶段研究编排流程
- 将复杂请求的 planning 产物结构化，并保存到当前工作区对应 session 的文件保存位置
- 保持 `union-search-plus` 作为补充来源 skill，而不是默认主链
- 降低用户交互负担，仅在真正需要时请求确认

## 2. 设计原则

- 简单问题优先快答，不强行进入深度调研流程
- 复杂问题先对齐研究范围，再进行大规模搜索
- 用户只在 planning 阶段做一次明确确认，后续由 agent 自动展开
- 研究计划需要可落盘、可复用、可审计
- 主链搜索优先，补充来源按需触发
- 最终结论必须附来源，搜索结果不等于事实验证

## 3. 方案对比

### 方案 A：简单直答 + 复杂双阶段规划 + 动态补源（推荐）

流程如下：

1. 判断请求是简单还是复杂
2. 简单请求直接搜索并回答
3. 复杂请求先基于内部知识与初步搜索产出 `json` 方案
4. 用户确认方案后，自动展开 `outline + source plan`
5. 执行搜索与抓取，必要时启用 `union-search-plus`
6. 汇总为报告

优点：

- 兼顾响应速度与调研质量
- 用户在真正大规模搜索前拥有控制权
- 与 `union-search-plus` 的职责边界最一致
- 允许执行中小幅纠偏，避免前置计划稍有偏差就整轮失效

缺点：

- skill 描述比当前版本更复杂
- 需要精心设计 planning JSON，避免和后续 outline 重复

### 方案 B：复杂请求单次确认后直接执行

流程如下：

1. 简单 / 复杂分流
2. 复杂请求一次性产出“方案 + 大纲 + 来源策略”
3. 用户确认后直接执行
4. 输出报告

优点：

- 交互最短
- skill 文本更简洁

缺点：

- `outline` 与来源策略容易过粗
- 不利于后续日志、回放与多 agent 协作
- 执行偏离计划时缺少清晰修订依据

### 方案 C：样本搜索后再二次确认

流程如下：

1. 复杂请求先产出初步方案
2. 执行小样本搜索
3. 基于样本结果再产出最终大纲与来源计划
4. 用户二次确认
5. 全量执行并输出报告

优点：

- 调研质量最高
- 对模糊需求更稳

缺点：

- 用户交互过重
- 普通复杂请求会感到繁琐
- 对 skill 和执行成本要求更高

### 结论

采用方案 A。它最适合当前项目状态，既保留复杂调研所需的前置确认，又避免多次重复审批；同时能清晰接入 `union-search-plus` 作为补充来源。

## 4. 新的职责边界

`research-union` 应定义为“研究编排 skill”，负责：

- 判断请求复杂度
- 组织复杂请求的 planning、outline、来源策略、执行与报告
- 在必要时向用户确认研究方案
- 协调主链搜索与补充来源 skill
- 输出带证据的研究结论

`research-union` 不负责：

- 替代简单搜索流程
- 默认直接触发所有补充来源
- 在未确认复杂方案前就进入大规模搜索
- 将 `union-search-plus` 内联成自己的默认搜索主链

`union-search-plus` 继续只负责：

- 在主链结果不足时补充来源
- 合并、去重、重排结果
- 为上层 research/report 提供更完整的证据池

## 5. 请求复杂度判定

### 5.1 简单请求

满足以下特征之一，通常走简单路径：

- 单点事实查询
- 只需要几个链接或一个简短摘要
- 单网页/单资料的摘要、提炼
- 不要求正式研究计划或结构化报告

### 5.2 复杂请求

建议采用“命中任意 2 条即进入复杂路径”的规则：

- 需要回答多个子问题
- 需要多来源交叉验证
- 涉及竞品对比、方案比较、趋势梳理、事实核验
- 用户明确要求“深入调研 / 全面梳理 / 系统研究 / 研究报告”
- 需要明确时间范围、地域范围、样本边界或排除项
- 预期输出是结构化报告，而不是一句话结论

### 5.3 判定后的处理

- 简单路径：直接搜索并回答，不使用 `ask_user`
- 复杂路径：必须先产出 planning JSON 并等待用户确认

## 6. 总体流程设计

### 6.1 简单路径

适用于快速检索任务：

1. 快速理解问题
2. 直接调用主链搜索工具
3. 按需抓取少量高价值页面
4. 输出简短结构化回答

特点：

- 不落 planning 文件
- 不进入复杂调研编排
- 不默认调用 `union-search-plus`

### 6.2 复杂路径

复杂请求固定为五阶段：

1. `step1_plan`
2. `step2_outline`
3. `step3_sources`
4. `step4_execute`
5. `step5_report`

其中只有 `step1_plan` 需要用户确认，后续阶段默认由 agent 自动推进。

## 7. Step1：Planning 设计

### 7.1 输入来源

planning 阶段基于以下信息生成：

- 当前用户需求
- agent 内部知识
- 少量初步搜索结果

这里的“初步搜索”只用于校准范围、术语和可行性，不是正式深度执行。

### 7.2 输出格式

复杂请求的 planning 必须生成可确认的 JSON，对话中向用户展示摘要，并将完整 JSON 保存到当前工作区对应 session 的文件保存位置。

建议字段如下：

```json
{
  "mode": "complex_research",
  "topic": "用户研究主题",
  "goal": "本次研究要回答的核心目标",
  "scope": {
    "time_range": "近3年",
    "region": "全球/中国/特定地区",
    "language": ["zh", "en"],
    "inclusions": ["纳入范围1", "纳入范围2"],
    "exclusions": ["排除范围1"]
  },
  "questions": [
    "子问题1",
    "子问题2",
    "子问题3"
  ],
  "depth_strategy": "先广后深 / 先官方后第三方 / 以事实核验优先",
  "expected_output": {
    "format": "report",
    "sections": ["结论摘要", "关键发现", "证据与来源", "分歧与限制"]
  },
  "source_plan": {
    "primary": ["serper_search", "fetch_url"],
    "supplement": ["union-search-plus if needed"]
  },
  "success_criteria": {
    "min_sources": 3,
    "min_evidence": 6,
    "coverage_target": 0.45
  },
  "confirmation_needed": [
    "范围是否准确",
    "是否补充或删除子问题",
    "时间范围是否需要调整"
  ]
}
```

### 7.3 用户交互

`ask_user` 只确认这三类内容：

- 研究范围是否准确
- 子问题是否需要增删
- 时间 / 地域 / 排除项是否需要调整

不在此阶段向用户暴露过多实现细节，避免把研究编排转成“让用户自己设计搜索系统”。

### 7.4 Planning 落盘规则

复杂请求的 planning 必须落盘，但默认写入当前工作区对应 session 的文件保存位置，而不是正式文档目录。

建议保存形式：

- `<session_workspace>/research/plan.json`

要求：

- 文件内容以 JSON 为主，便于后续阶段和其他 agent 读取
- 文件路径需进入日志，便于调试
- 后续 `outline / sources / execution / report` 可引用该文件，而不是完全依赖上下文记忆

## 8. Step2：Outline 设计

用户确认 planning 后，agent 自动生成正式调研大纲。

每个子问题应明确：

- 需要回答什么
- 需要哪些类型的证据
- 是否涉及时间维度、对比维度或定义澄清

`outline` 的职责是把“研究意图”转成“执行骨架”，不再二次请求用户审批，除非用户主动要求复核。

可选输出结构：

```python
class ResearchOutline:
    topic: str
    sections: list[str]
    questions: list[str]
    evidence_requirements: list[str]
    notes: list[str]
```

## 9. Step3：来源计划设计

来源计划必须与 `outline` 绑定，而不是只写一句“会搜索”。

建议组织方式：

- 子问题 -> 优先来源类型 -> 备用来源 -> 是否需要抓取正文

默认策略：

- 主链始终优先：`serper_search + fetch_url`
- 优先官方文档、原始公告、权威媒体、可靠研究机构
- 只有在主链覆盖不足、证据不足，或用户明确要求补充来源时，才调用 `union-search-plus`

这一步的输出可以是机器可读结构，也可以写入运行时中间文件，供执行阶段复用。

## 10. Step4：执行设计

执行阶段按以下顺序进行：

1. 为各子问题生成查询
2. 主链搜索
3. 抓取高价值页面
4. 评估来源覆盖与证据充分性
5. 必要时调用 `union-search-plus`
6. 合并、去重、重排结果

### 10.1 小幅纠偏

执行阶段允许“一次小幅纠偏”，例如：

- 原子问题过宽，需要收窄
- 某子问题证据不足，需要改关键词
- 某来源类型质量差，需要替换为更权威来源

以下情况应重新请求用户确认：

- 研究范围明显扩大
- 时间范围或地域范围明显变化
- 需要新增用户未授权的高成本 / 高风险补充搜索

### 10.2 与 `union-search-plus` 的衔接

固定规则：

- 先主链，后补充
- 先 `preferred`，再 `all`
- 若用户拒绝审批或补充分支失败，保留主链继续完成任务
- 不得因为补充分支失败而中断最终回答

## 11. Step5：报告设计

最终报告建议固定为五段：

1. 结论摘要
2. 关键发现
3. 证据与来源
4. 分歧 / 不确定点
5. 限制与补充说明

硬性要求：

- 关键结论必须可追溯到来源
- 搜索到不等于验证过，措辞需谨慎
- 若补充分支未执行、失败或用户拒绝审批，必须明确说明
- 若不同来源存在冲突，应保留分歧，不强行合并为单一结论

## 12. 对 `research-union` skill 文案的改动建议

建议重写 `SKILL.md` 的核心结构：

1. 明确自身是研究编排 skill
2. 增加简单 / 复杂双路径
3. 定义复杂度判定标准
4. 定义复杂路径五阶段
5. 明确 planning JSON 的结构和用户确认点
6. 明确 planning 需要保存到 session 工作区
7. 明确 `union-search-plus` 只作补充来源
8. 明确报告输出结构与禁止事项

## 13. 对 `search-agent` 提示词的改动建议

`search-agent` 的 `system_prompt` 应同步收敛到以下逻辑：

- 先判断简单 / 复杂
- 复杂请求优先使用 `research-union`
- 简单请求直接使用搜索工具完成
- 不要把复杂流程临时内联拼接
- 不要在主链足够时默认触发补充来源
- 报告和回答都要坚持“先结论，后依据，再说明限制”

## 14. 测试与验证建议

建议新增或更新以下验证：

- skill 加载测试：确认 `research-union` 可被正确加载
- 提示词 / skill 路由测试：简单请求不进入复杂路径，复杂请求会进入 planning
- 复杂 planning 格式测试：确认生成 JSON 结构满足字段要求
- session 工作区落盘测试：确认 `plan.json` 写入当前会话对应保存位置
- 进程内 e2e：验证复杂请求链路为 `planning -> ask_user -> outline -> sources -> execute -> report`
- 补充分支回归：当主链不足时触发 `union-search-plus`，当用户拒绝或失败时正确降级

## 15. 风险与注意事项

- 如果复杂度判定过宽，用户会感到交互负担增加；因此必须坚持“命中任意 2 条”原则
- 如果 planning JSON 字段过多，模型容易输出不稳定；应保持字段数量克制
- 如果 `outline` 与 planning 描述重复，用户会感觉流程冗余；因此后续阶段默认不再重复确认
- 如果 planning 不落盘，执行阶段会更依赖上下文，降低可调试性和可协作性
- 如果补充来源触发条件不清晰，`union-search-plus` 会被误用为默认搜索主链

## 16. 结论

本设计建议将 `research-union` 升级为“简单直答 + 复杂五阶段研究编排”的 skill：

- 简单请求直接答
- 复杂请求先基于内部知识与初步搜索形成 planning JSON
- 用户确认后自动展开 `outline + source plan`
- 执行中允许一次小幅纠偏
- 最终输出带来源、分歧与限制的报告

同时，复杂请求的 planning 必须保存到当前工作区对应 session 的文件保存位置，作为后续执行与调试的统一依据；`union-search-plus` 继续保持“补充来源”角色，不升格为默认主链。
