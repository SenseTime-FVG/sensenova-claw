# research-union（简单直答 + 复杂调研编排）

这是一个“研究编排 skill”，不是单纯的搜索 skill，也不是补充搜索源 skill。

它的职责是：

- 判断用户需求是简单 research 还是复杂 research
- 在复杂场景下，先形成可确认的研究方案
- 让用户只在 planning 阶段做一次关键确认
- 协调主链搜索与补充来源
- 将结果整理为带证据的最终回答或报告

它不负责：

- 在简单问题上强行走完整 deep research 流程
- 在未确认复杂方案前直接进入大规模搜索
- 把 `union-search-plus` 变成默认主链
- 把原始搜索结果原样倾倒给用户

`union-search-plus` 的定位保持不变：

- 它是补充来源 skill
- 只在主链结果不足或用户明确要求补充来源时使用

## 何时使用

以下情况优先使用本技能：

- 用户明确要求“深度调研”“全面梳理”“系统研究”“研究报告”
- 问题涉及多来源对比、竞品/方案比较、趋势分析、事实核验
- 问题不是单点事实，而是需要拆成多个子问题才能回答
- 用户强调结果要附依据、来源和分歧说明

以下情况通常不要使用本技能：

- 单条事实问题，直接搜索即可回答
- 只需要几个链接或一个简短网页摘要
- 本地文件或已有上下文就足以回答的问题

## 路径选择

先判断用户需求属于哪条路径。

### `simple path`

适用于以下情况：

- 单点事实查询
- 少量链接、快速摘要
- 用户没有要求研究计划、结构化报告或多来源交叉验证

执行原则：

- 不使用 `ask_user` 询问研究方案
- 直接使用搜索工具完成
- 默认不落 planning 文件
- 先给结论，再给依据

### `complex path`

当以下信号命中任意 2 条时，进入复杂路径：

- 需要回答多个子问题
- 需要多来源交叉验证
- 涉及竞品对比、方案比较、趋势梳理、事实核验
- 用户明确要求“深入调研 / 全面梳理 / 研究报告”
- 需要明确时间范围、地域范围、样本边界或排除项

复杂路径固定为五阶段：

1. `step1_plan`
2. `step2_outline`
3. `step3_sources`
4. `step4_execute`
5. `step5_report`

其中只有 `step1_plan` 需要用户确认。后续步骤默认自动推进，除非研究范围发生明显变化。

## `complex path` 详细流程

### `step1_plan`：先形成 research planning JSON

先结合以下信息形成初步方案：

- 用户原始问题
- 当前上下文中的内部知识
- 少量初步搜索结果

这里的“初步搜索”只用于判断术语、范围和可行性，不是正式深度执行。

必须产出一个可确认的 JSON 方案。建议至少包含：

```json
{
  "mode": "complex_research",
  "topic": "用户研究主题",
  "goal": "本次研究要回答的核心目标",
  "scope": {
    "time_range": "近3年",
    "region": "全球/中国/特定地区",
    "language": ["zh", "en"],
    "inclusions": ["纳入范围1"],
    "exclusions": ["排除范围1"]
  },
  "questions": [
    "子问题1",
    "子问题2",
    "子问题3"
  ],
  "depth_strategy": "先广后深 / 先官方后第三方 / 以事实核验优先",
  "expected_output": {
    "format": "report",
    "sections": ["结论摘要", "关键发现", "证据与来源", "分歧与限制"]
  },
  "source_plan": {
    "primary": ["serper_search", "fetch_url"],
    "supplement": ["union-search-plus if needed"]
  },
  "success_criteria": {
    "min_sources": 3,
    "min_evidence": 6,
    "coverage_target": 0.45
  }
}
```

向用户展示时：

- 用简洁中文概括核心范围
- 明确告诉用户：现在确认的是研究范围、子问题、时间/地域/排除项
- 不要把过多内部实现细节甩给用户

使用 `ask_user` 收集确认结果。只确认这些内容：

- 范围是否准确
- 子问题是否需要增删
- 时间 / 地域 / 排除项是否需要调整

如果用户确认：

- 使用 `write_file` 将完整 planning JSON 写入当前 session 工作区的 `research/plan.json`
- 这个文件是后续步骤的统一依据
- 在后续步骤中优先引用该文件，而不是完全依赖上下文记忆

如果用户要求调整：

- 根据反馈修订 planning JSON
- 再次确认
- 未确认前不要进入大规模搜索

### `step2_outline`：生成正式大纲

在 planning 确认并落盘后，自动生成正式 `outline`。

每个子问题都应明确：

- 要回答什么
- 需要什么类型的证据
- 是否涉及时间维度、定义澄清或对比维度

这一阶段默认不再向用户发起第二次确认，除非研究范围明显变化。

### `step3_sources`：确定来源策略

来源计划必须与 `outline` 绑定，而不是笼统说“会搜索”。

默认规则：

- 主链始终优先：`serper_search + fetch_url`
- 优先官方文档、原始公告、权威媒体、可靠研究机构
- 当主链覆盖不足、证据不足，或用户明确要求补充来源时，再调用 `union-search-plus` skill 补充搜索来源

不要一开始就默认启用 `union-search-plus`。

### `step4_execute`：按计划执行

执行顺序：

1. 为各子问题生成查询
2. 执行主链搜索
3. 对高价值页面调用 `fetch_url`
4. 评估覆盖情况与证据充分性
5. 必要时调用 `union-search-plus`
6. 合并、去重、重排结果

执行过程中允许一次“小幅纠偏”，例如：

- 子问题过宽，需要收窄
- 关键词不够准，需要微调
- 某类来源质量差，需要换成更权威来源

以下情况应重新请求用户确认，而不是自行扩大范围：

- 研究范围明显扩大
- 时间范围或地域范围明显变化
- 需要新增用户未授权的高成本补充搜索

### `step5_report`：整合为报告

最终输出建议固定为以下结构：

1. 结论摘要
2. 关键发现
3. 证据与来源
4. 分歧 / 不确定点
5. 限制与补充说明

输出要求：

- 先结论，后依据
- 关键结论必须附来源链接
- 若存在冲突信息，必须明确标注分歧
- 若补充分支未执行、失败或用户拒绝审批，必须明确说明

---

## 与 `union-search-plus` 的衔接

复杂路径中，`union-search-plus` 只作为补充来源使用。

固定顺序：

1. 先主链
2. 不足时再补充
3. 先 `preferred`
4. 必要时才升级到 `all`

若补充分支失败或用户拒绝审批：

- 继续使用已有主链结果完成任务
- 在最终回答中明确说明限制

不得因为补充分支失败而让整个 research 停摆。

## 禁止事项

- 不得在简单问题上强行走复杂流程
- 不得在 planning 未确认前就进入大规模搜索
- 不得把 `union-search-plus` 当成默认主链
- 不得编造来源
- 不得把“搜索到”写成“已验证”
- 不得省略关键结论的来源链接
- 不得把 planning 只放在上下文里而不落盘

## 执行检查清单

结束前确认：

- 是否先判断了 `simple path` 还是 `complex path`
- 如果是复杂路径，是否生成并确认了 planning JSON
- 是否将 planning 写入当前 session 工作区的 `research/plan.json`
- 是否在 `step2_outline`、`step3_sources`、`step4_execute`、`step5_report` 中按顺序推进
- 是否优先使用主链，只在必要时补充来源
- 最终答案是否保留来源、分歧和限制说明
