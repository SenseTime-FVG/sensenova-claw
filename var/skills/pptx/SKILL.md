# PPT 编排技能

这是当前的简版 PPT 工作流。当前默认交付物是 HTML 幻灯片 deck，而不是 `.pptx` 文件。

当任务需要完成以下事情时，使用本技能：

- 理解用户的 PPT 需求
- 收集或解析支撑材料
- 将信息汇总成一份可执行 brief
- 在需要时抽取整套 deck 的统一风格
- 将大纲生成交给下游 skill
- 将图片检索与筛选交给下游 skill
- 将逐页 HTML 生成交给下游 skill

如果要走完整的未来流程，可以参考同目录下的 [design.md](design.md)。当前先坚持简版路径。

## 默认范围

本技能适用于：

- 根据用户 query 新建一套 PPT/deck
- 根据上传文档或链接整理一套 deck
- 在目标页数固定的前提下，输出逐页 HTML 形式的演示稿

本技能当前不负责：

- 假装已经支持 `.pptx` 导出
- 复杂模板逆向解析和模板继承
- 强动画、强动效的页面生成

## 工作流

### 1. 任务标准化

先提取或推断以下信息：

- 主题
- 受众
- 演示目标
- 目标页数
- 风格偏好
- 必须覆盖的部分
- 用户提供的文件、链接和参考资料

默认语言规则：

- `brief.md`、`outline.json` 中的页面标题与正文内容、`style.json` 中的自然语言说明、以及最终 HTML 页面里的用户可见文本，默认都要与用户 query 的语言保持一致。
- 只有当用户明确要求使用另一种语言时，才切换输出语言。
- 固定的 JSON 键名、工具约定字段名、代码字段名、枚举值可以保持英文；但面向用户阅读的自然语言内容不要无故切到英文。
- 如果用户用中文提问，就不要产出英文标题、英文大纲和英文页面文案；如果用户用英文提问，也不要默认切成中文。

如果页数没给，但用户显然要一套 deck，可以先提出一个合理页数并说明这个假设。

如果核心意图仍不明确，只对阻塞生成的缺口追问。

### 2. 规划信息收集

按需选择工具，不要全部都用：

- `read_file`：读取用户提供的文档文件
- `serper_search`：补充外部事实、背景和案例
- `fetch_url`：抽取指定网页的有效内容

如果调研或资料消化很重，可以分阶段处理，避免主上下文被污染。

在创建 PPT 相关任务时，必须把工作包写得足够像一个 PPT 任务。

任务描述里在相关时应明确包含：

- `PPT`、`slide deck`、`slides`、`presentation`、`HTML slides` 这类词
- 当前阶段名称：`style extraction`、`outline generation`、`HTML generation`
- 关键产物名：`brief.md`、`outline.json`、`style.json`、`page_XX.html`
- 最终输出目录，也就是 deck 目录

不要把任务写成泛化的”生成 html 文件”。应写成更明确的 PPT 工作包，例如”根据 outline.json 和 style.json 生成 PPT HTML slides，并写入指定 deck 目录”。

### 3. 生成汇总 brief

在写大纲前，先整理出一份内部 brief，至少包含：

- deck 的核心叙事
- 受众和决策场景
- 关键证据或支撑事实
- 建议的叙事结构
- 这套 deck 的基础视觉方向
- 约束、假设和缺失项

brief 必须足以支撑后续决策，并且要和最终 deck 产物保存在同一个输出目录中。

brief 的简版格式必须固定为 3 部分：

1. 整体内容汇总
2. 叙事大纲简述
3. 分页内容汇总

补充规则：

- `brief.md`、`style.json`、`outline.json` 和最终 `page_XX.html` 必须放在同一个 deck 目录下。
- `整体内容汇总` 的字数不得少于 `400 * 总页数`。
- `分页内容汇总` 必须按最终页面顺序逐页概述，且和最终大纲保持一致。
- `brief.md` 中面向用户阅读的自然语言内容默认必须与用户 query 的语言一致；除非用户明确指定其他语言，否则不要切成英文或中英混写。
- 不要把原始调研材料原样堆进 brief。
- `brief` 阶段只有在目标 `brief.md` 已真实写入磁盘后才算完成，不能只在 `subtask_result` 中口头声称“已生成”。
- 在主线程把 `brief` 标记为 `completed` 之前，必须先验证目标文件存在且可读。
- 如果子任务负责撰写 `brief.md`，子任务说明中必须明确写出目标文件路径，且子任务返回结果里要包含该文件已落盘的可验证事实。
- 如果主线程读取 `brief.md` 失败、找不到文件、或文件内容明显不完整，不要沿用“已完成”的判断；应继续补写、重试或把该工作包标记为未完成。
- 不要为了补齐流程临时写一个过短的占位 brief；如果必须补写，也必须满足当前页数对应的字数与三段式结构要求。

### 4. 按需抽取整套 deck 的风格

如果任务里包含风格参考图、明确的视觉要求，或者报告本身已经带有足够的风格线索，就先执行 `ppt-style-extract` 风格提取流程。

它负责产出整套 deck 的风格规范，通常保存为 `style.json`。

如果没有足够的真实风格信号，可以跳过这一步，后续页面设计保持克制和保守。

阶段闸门要求：

- 在进入下一阶段前，必须先验证上一阶段的目标产物已落盘、可读取且结构合法。
- `style.json` 必须遵循 `ppt-style-extract` 约定的 canonical schema，不得自定义简化版 `style.json`，也不要临时改用另一套键名。
- 如果 schema 不符合约定，当前阶段不得标记为完成，必须先修正产物再继续。

### 5. 交给大纲生成 skill

在开始规划页面前，先执行 `ppt-outline-gen` 大纲生成流程。

它负责：

- 严格匹配目标页数的大纲生成
- 页面类型和布局规划
- 图片分配和补图建议
- 大纲 JSON 结构

阶段闸门要求：

- 主线程在把大纲阶段标记为完成前，必须先验证 `outline.json` 已写入 deck 目录、能被读取、且结构符合 `ppt-outline-gen` 约定。
- 不要接受“更短但不兼容”的替代格式；如果子任务私自删字段、改字段名或压扁结构，应视为未完成。
- 如果某页没有 `needed_pictures`，就不要在该页预留图片槽位，也不要选择 `image_left_text_right`、`text_left_image_right` 等图片导向布局来硬塞空图位。

### 6. 交给 HTML 生成 skill

如果大纲中存在 `needed_pictures`，在进入 HTML 生成前，先执行 `ppt-image-selection` 图片选择流程。

它负责：

- 基于 `needed_pictures` 构造搜图 query
- 调用 `image_search` 获取候选图片
- 按页面语义和图片槽位做筛选
- 只保留可直接稳定下载的候选图
- 将最终通过的图片逐张下载到 deck 目录的 `images/` 子目录
- 产出 `image_search_results.json` 和 `image_selection.json`

在这一步里，下载必须带失败退出和超时，且要逐张校验本地文件是否真实存在、非空、可作为图片打开。
如果某个候选 URL 下载失败，不要硬用远程链接继续推进；应切换候选或将该槽位标为 `unresolved`。

阶段闸门要求：

- `image_selection.json` 必须使用 `ppt-image-selection` 约定的结构化格式，不得自定义简化版 `image_selection.json`。
- 只有当 `image_search_results.json`、`image_selection.json` 和 `images/` 本地图片目录都已准备好，且 `selected_image.local_path` 指向真实存在的本地图片时，才能进入 HTML 阶段。
- 如果某些槽位最终没有成功下载图片，也必须在 `image_selection.json` 中显式记录失败状态和原因，不能静默跳过。

如果大纲中没有 `needed_pictures`，可以跳过这一步。

### 7. 交给 HTML 生成 skill

在大纲稳定后，再执行 `ppt-html-gen` HTML 生成流程。

它负责：

- 单页 HTML 生成
- 页面尺寸和安全区约束
- 图片、图表、表格的实现规则
- 最终逐页文件输出

阶段闸门要求：

- 只有在 `brief.md`、`outline.json`、以及按需存在的 `style.json`、`image_selection.json` 都通过验收后，才允许生成 `page_XX.html`。
- 如果上游任何一个必要产物不存在、不可读、或 schema 不合法，不要口头声称“可以先继续”，应先回到对应阶段修正。

### 8. 结果打包

简版 deck 产物通常应包含：

- `brief.md`
- 使用了风格抽取时的 `style.json`
- `outline.json`
- 存在补图时的 `image_search_results.json`
- 存在补图时的 `image_selection.json`
- 存在补图时的 `images/` 本地图片目录
- `page_01.html`、`page_02.html` ...

这些产物必须全部保存在同一个 deck 目录下。

默认一页一个 HTML 文件。除非用户明确要求，不要把整个 deck 压成一个超大的单 HTML。

最后用 `display_result` 或等价结果工具把生成文件返回给用户。

## 质量要求

- 严格匹配目标页数。
- 保证整套 deck 的叙事连续性。
- 优先保证结构清晰，而不是过早追求复杂视觉。
- 在生成页面之前，先保证 brief 的三段式结构完整且达标。
- `brief.md` 必须是可直接交付和复用的真实文件，而不是只存在于 agent 对话或 `subtask_result` 里的摘要。
- 所有面向用户阅读的自然语言内容默认必须与用户 query 的语言一致，除非用户明确要求使用另一种语言。
- 如果大纲包含补图需求，先完成图片检索和筛选，再进入 HTML 阶段。
- 如果筛图阶段无法把图片成功下载到本地，不要把远程 URL 当最终交付图片资产。
- 优先选择可稳定下载的图片资源，不要把社交平台代理图、临时抓取图或明显依赖页面脚本的 URL 当最终资产。
- 只能使用真实拿到的信息，不要虚构数据、案例或引用。
- 如果素材不足以支撑目标页数，应在 brief 中说明限制，并调整大纲密度，而不是臆造内容。
- 如果 brief 信息充足，正文页应适当承载更多内容，不要过度压缩成空洞的“大标题 + 一句话”页面。
